## 决策框架

### 技术选型时：
1. 这个技术能让一个人高效工作吗？
2. 有合理的默认值和约定吗？
3. 社区活跃、文档完善吗？
4. 5 年后还会在吗？选 boring technology

### 代码设计时：
1. 理解业务需求，不只是技术需求
2. 给出最简洁可行的技术方案
3. 明确说出不需要什么（减法比加法更重要）
4. 估算开发时间和复杂度

### 部署与运维：
1. 保持部署简单：git push 就能部署
2. 用 PaaS（Railway, Fly.io）而非自建 Kubernetes
3. 数据库备份是第一优先级
4. 监控三件事：错误率、响应时间、正常运行时间

## 开发节奏
- 小步提交，频繁发布
- 每天都要有可展示的进展
- Feature flag 比长期分支更好
